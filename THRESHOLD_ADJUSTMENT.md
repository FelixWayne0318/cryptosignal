# v7.2闸门阈值调整说明

## 问题诊断

### 症状
- 扫描383个币种，发现172个基础信号
- 但v7.2五道闸门**全部拒绝**，最终0个发送
- 日志显示"扫描无结果"

### 根本原因

**v7.2统计校准在冷启动模式下过于保守**

数学分析：
```
典型信号特征：
- 置信度：Conf = 27
- F因子：F = -40（资金流出）
- I因子：I = 50

概率计算（启发式公式）：
P_base = 0.45 + (27/100) * 0.23 = 0.5121

F因子惩罚（F < -30）：
P_final = 0.5121 - 0.02 = 0.4921

Gate4阈值检查：
0.4921 < 0.50 ❌ 被拒绝！
```

**问题所在**：
1. 大多数币种F因子为负（市场资金流出）
2. 负F因子触发 -0.02 惩罚
3. 本就接近临界的概率被压到0.50以下
4. Gate4 (P_min=0.50) 拒绝所有信号

## 解决方案

### 调整1: 降低Gate4概率阈值

**修改**：`config/signal_thresholds.json`

```json
"gate4_probability": {
  "P_min": 0.45  // 从 0.50 降到 0.45
}
```

**影响**：
- 允许概率 0.45-0.50 的信号通过
- 配合冷启动启发式公式（P = 0.45 + Conf/100 * 0.23）
- 信号通过率预计提升至 30-50个/扫描

### 调整2: 放宽F因子要求

**修改**：`config/signal_thresholds.json`

```json
"gate2_fund_support": {
  "F_min": -50  // 从 -15 降到 -50
}
```

**影响**：
- 允许资金流出但技术形态好的信号
- F ∈ [-50, -15] 的信号现在可以通过
- 适应当前市场环境（整体资金流出）

### 调整3: 同步AntiJitter阈值

**修改**：`config/signal_thresholds.json`

```json
"AntiJitter参数": {
  "min_probability": 0.45  // 从 0.50 降到 0.45
}
```

**影响**：与Gate4保持一致，避免二次过滤

## 预期效果

### 信号通过率
```
修改前：172个基础信号 → 0个通过v7.2 → 0个发送
修改后：172个基础信号 → 30-50个通过v7.2 → 发送Top 1
```

### 发送频率
- 每5分钟扫描一次
- 发送Top 1最优信号（按confidence_adjusted排序）
- 预计每次扫描发送1个信号

### 信号质量
- **仍保留**：Gate1（数据质量）、Gate3（EV>0）、Gate5（I×Market）
- **放宽**：Gate4（概率）、Gate2（F因子）
- 质量下降：P从0.50降到0.45（-10%）
- 风险：可控（仍需通过4/5道门槛）

## 风险控制

### 降低阈值的风险
1. **假阳性增加**：P=0.45-0.50的信号可能质量略低
2. **回撤可能增加**：更多信号意味着更多交易

### 缓解措施
1. **保留多道闸门**：仍需通过5道门槛中的4道
2. **Top 1策略**：每次只发送最优信号
3. **AntiJitter**：1小时冷却期，避免重复信号
4. **统计反馈**：积累30+个信号后自动校准阈值
